# 章ごとの書き方ガイド

各章の方針・必須要素・書いてはいけないこと。テンプレ([`templates/`](../templates/))の TBD 箇所を埋めるときはこれを読む。

## 共通方針

### トーン

- **技術ブログ調**。論文ではないので、です・ます調と「〜と思う」「〜という発見だった」のような主観的トーンを混ぜてよい
- ただし数値結果に主観形容(「めちゃくちゃ早い」「ものすごい」)を付けない。数値は数値で語らせる

### 構造の鉄則

検証結果(03)・創発観察(04)・頑張った点(05)は、必ず以下の構造で書く:

```
パラメータ X を [A → B] に変えた → 創発 Y が [P → Q] に変化した
  - 観察された会話例: ...
  - 数値裏付け: ...
  - 解釈: ...
```

技術スタックの羅列(「Ollama を使いました」「Qwen3 を選びました」)は **02_システム構成** にだけ書く。他の章には書かない。

### 失敗の書き方

失敗・没案を載せる目的は「謙虚さアピール」ではなく「**意思決定の解像度を見せる**」こと。以下のフォーマットで:

```markdown
### XX を選ぶ前に Y を試した

- 当初の仮説: ...
- 実際にやってみたこと: ...
- うまくいかなかった理由: ...(具体的に。「なんとなく合わなかった」は禁止)
- 切り替えた判断: ...
```

「失敗しました」だけだと審査員に何も伝わらない。**失敗から何を学んだ結果、いまの選択になっているか**を必ず書く。

## 00_表紙

- LaTeX のべた書きで構わない(Pandoc Markdown は LaTeX を素通しする)
- 著者名・GitHub・日付・バージョンは [`metadata.yaml`](../templates/metadata.yaml) と一致させる
- ロゴや図を入れたい場合は `assets/` に置いて `\includegraphics` で参照

## 01_思想と背景

**目的**: 「何を観察したかったか」を 1 文で言い切る。

- 中核の問い(LLMモデル/男女比/人種/年齢 → 創発の差)を最初の太字 1 行で
- 参考実装からの差分を 3 点以内で
- 3 階層モデルは表で。文章で延々と説明しない

**やってはいけない**: 「LLM とは...」のような前提解説で文字数を稼ぐこと。審査員は LLM を知っている。

## 02_システム構成

**目的**: 「これで動きます」が伝わればよい。読者がもう一回作ろうと思ったら作れる程度の情報量。

- アーキ図(Mermaid → Pandoc 経由で PDF に埋め込み or PNG 化)
- ハードウェア表
- 採用 LLM 表
- 5 種類の jsonl ログ表
- abliterated を選んだ要約 1 段落(詳細は 05 章へリンク)

**やってはいけない**: コード断片を貼り付けて埋めること。コードはリポジトリで読める。

## 03_検証結果

**目的**: 数値で創発の差を見せる。

- 性能サマリ表(構成 × 並列度)
- 比較軸ごとに節を切る(モデル / 男女比 / 人種・年齢)
- 各節の最後に必ず「数値だけでは見えない部分」を 1 段落書く
- グラフは matplotlib で生成 → `assets/figures/*.png` → Markdown で埋め込み

**やってはいけない**: 「TBD」のまま提出すること。値が無い場合は **「N=3 の予備実験のため有意性は次節で議論」** のように明示する。

## 04_創発観察

**目的**: ログ抜粋で「これが創発だ」を見せる。

- 6 つの観察軸(嘘・派閥・通貨・反乱・合意形成・発言権)それぞれに節を切る
- 各節の冒頭に **観察日時 / エージェント ID / シナリオ** を記す
- 抜粋はコードブロック(```text```)で生のまま貼る。整形しない
- **解釈**は抜粋の後に必ず 1 段落書く

**やってはいけない**:
- チェリーピックで「派手な run」だけ拾うこと → 反例も併記
- 抜粋を編集・要約すること → 原文を保つ
- 個人攻撃や差別表現を「事故的に消す」こと → 観察対象として残す(研究目的の合意済みである旨を冒頭に明示)

## 05_頑張った点

**目的**: 「何を考えて、何を試したか」が伝わる章。技術スタックの自慢ではない。

- 比較設計の枠組み(4 軸を独立に動かせる config 化)を最初に
- 失敗・没案を 3〜5 件、上記のフォーマットで列挙
- 最後に「技術的挑戦」で締める

**やってはいけない**: 「徹夜しました」「大変でした」のような労働量アピール。

## 06_未解決課題

**目的**: 「ここまでは解けて、ここから先は解けていない」を率直に。

- 各課題に「**改善案**」を必ずセットで書く(問題提起だけで終わらせない)
- 統計的有意性、ペルソナのバイアス、VRAM 由来の制約、創発指標の主観性、動画の網羅性など
- 隠さない・盛らない

**やってはいけない**: 「もっと時間があれば」だけで締めること。**何にどれだけ時間が必要か**を具体に。

## 07_今後の展望

**目的**: 1 階を入れ替える拡張案を見せて、フレームワークの汎用性を示す。

- 細胞 / 虫 / 人(現行) / ウイルスの表(1 階・2 階・3 階・観察したい創発)
- 規模拡張(100 → 1000、マルチノード)
- 公開プラン(リポジトリ URL、議事録ごと公開する立場)

**やってはいけない**: 「いつかやります」の羅列。優先度を 1 つは付ける。

## 章再生成のチェックリスト

特定章を再生成するときは:

1. `build/chapters/NN_*.md` の編集を一度コミット(レビューの履歴として残す)
2. 必要なら `python scripts/collect_assets.py --runs runs/<新ts>` で素材を更新
3. `.\build_pdf.ps1 -Chapters NN` で対象章だけ再生成
4. 生成された PDF を確認 → OK なら `output/` を更新

> **Why:** 章単位で差し替え可能であることはユーザの強い要望。検証ログが増えていく前提で、03 章と 04 章は提出ギリギリまで動く想定。
