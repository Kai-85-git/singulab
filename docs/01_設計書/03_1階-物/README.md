# 03_1階(物) 設計

オフィスの物理空間モデル。エージェントが占有・移動する「場」の定義。

> 本書は [2026-04-24 ミーティング §5.1](../../02_ミーティング/2026年4月/02_議事録/2026年4月24日.md) の決定事項に基づく。
>
> ⚠️ **位置付け**: たたき段階であり、実装前のレビュー・議論・上書きを前提とする。

## 1. 議事録からの決定

| 項目 | 方針 |
| --- | --- |
| 設定する場 | **オフィス**(会議室を内包する前提) |

> 1階は「物」。物理的に存在する場のみを定義する。環境パラメータ(景気等)は [2階](../04_2階-環境/)、世界のルールは [3階](../05_3階-物理-重力/) で扱う。

## 2. 空間モデル

### 2.1 採用する表現

- **2D 連続座標**を基本とし、`place` 単位で区画を持つ
- 参考実装 [2d-multi-places-simulation-on-fire-public](../../10_共有資料/2d-multi-places-simulation-on-fire-public/) の `PlaceConfig(center, half_size, capacity)` を踏襲
- 各 place はその中心点・半径・収容人数を持ち、エージェントはその矩形に入ったとき `in_place` 扱い

### 2.2 コア構造(案)

```yaml
places:
  - id: main_office
    center: [0.0, 0.0]
    half_size: [10.0, 10.0]
    capacity: 100
    role: "業務エリア"
  - id: meeting_room
    center: [5.0, 5.0]
    half_size: [2.0, 2.0]
    capacity: 10
    role: "会議室"
```

### 2.3 場の内訳

| place | 目的 | 収容 |
| --- | --- | --- |
| main_office | 通常業務・廊下・立ち話 | 集団サイズ全員 |
| meeting_room | 会議・意思決定の場 | 10 人(議事録どおり「会議室は 10 人が自然」) |

- **小集団(2〜10人)**: main_office + meeting_room。全員が会議室にも入れる
- **大集団(100人)**: main_office + meeting_room。会議室は 10 人までなので「誰が呼ばれるか」も観察対象になる(=発言権の創発の材料)

### 2.4 エージェントの位置と移動

- `pos: (x, y)` を保持
- 移動は 4 方向 + stay(参考実装準拠)
- `in_place` 判定で現在いる place を特定(参照: [06_システム設計/05_シミュレーションループ](../06_システム設計/05_シミュレーションループ/))

## 3. 4 象限(都心 / 地方 × 大企業 / スタートアップ)との関係

**1階は場のみ**。4 象限は以下で表現する:

| 軸 | 表現場所 |
| --- | --- |
| 都心 / 地方 | エージェントのペルソナプロンプト(「あなたは◯◯にいる」) |
| 大企業 / スタートアップ | 集団サイズ + ペルソナプロンプト |
| 景気 | [2階(環境)](../04_2階-環境/) |

物理モデル自体は全象限で共通とし、象限差はプロンプトと集団サイズで吸収する。

## 4. 実装方針

- `config.yaml` の `places` セクションで定義
- [06_システム設計/02_モジュール構成](../06_システム設計/02_モジュール構成/) の `World` モジュールが place の管理を担う
- エージェント追加時は初期 `pos` と所属 place を指定

## 5. スコープ外(MVP では扱わない)

- オフィス「間」の階層化(部署・フロア分け)
- 屋外・通勤・自宅
- 3D 空間
- 動的な place(壁が動く・会議室が増減する等)
