# 04_2階(環境) 設計

物理空間に被さる環境パラメータ。**景気のみ軽く付与**、他は創発として観察する。

> 本書は [2026-04-24 ミーティング §5.2](../../02_ミーティング/2026年4月/02_議事録/2026年4月24日.md) の決定事項に基づく。
>
> ⚠️ **位置付け**: たたき段階。

## 1. 議事録からの決定

| 項目 | 設定方針 | 理由 |
| --- | --- | --- |
| 景気(良い/悪い) | **軽く付与** | 全体に緩く効く前提条件として |
| 時間的プレッシャー | **設定しない** | 創発として観察したい |
| 情報の流通量 | **設定しない** | 創発として観察したい |
| 企業文化の雰囲気 | **設定しない** | 大企業/スタートアップに固定化されてしまうため |

## 2. 設定する: 景気

### 2.1 表現

- 値: `"good" | "bad" | "neutral"` の enum
- プロンプトへの注入: 1〜2 文の短文(命令は避ける、状態記述のみ)
  - 例(good): 「現在、景気は良い。」
  - 例(bad): 「現在、景気は悪く、予算・雇用に緊張感がある。」
  - 例(neutral): 「現在、景気は可もなく不可もない。」

### 2.2 象限ごとのデフォルト(仮)

|  | 地方 | 都心 |
| --- | --- | --- |
| 大企業 | neutral | neutral |
| スタートアップ | bad(余裕がない) | neutral |

- スタートアップ × 地方: 景気の悪影響を受けやすいことを反映し `bad` を仮置き
- 比較実験時はこの値を振って挙動差を見る

### 2.3 プロンプト注入位置

- エージェントの **system prompt 末尾** に追加
- 景気の変化はステップ途中で変わらない前提(1 シミュレーション内で固定)

## 3. あえて設定しない(= 観察する)

### 3.1 時間的プレッシャー
「締切まで◯◯時間」を与えない。エージェント同士の相互作用で「急いだ方が良さそう」等が **自発的に立ち上がるか** を観察する。

### 3.2 情報の流通量
「今日は情報がよく流れる」等を与えない。[3階](../05_3階-物理-重力/) のコミュニケーション到達距離と偶然の隣接によって流通量が決まる。

### 3.3 企業文化の雰囲気
「トップダウン」「フラット」「飲み会が多い」等の雰囲気を与えない。エージェントのやりとりから **自然と文化が立ち上がる** ことを観察する。ここを設定すると大企業/スタートアップに固定化されてしまう。

## 4. 実装方針

### 4.1 Config

```yaml
environment:
  economy: "good"  # "good" | "bad" | "neutral"
```

### 4.2 モジュール責務

- 2階は Simulation Orchestrator の「環境モジュール」として独立
- エージェント入力時、景気文を system prompt に挿入するだけのシンプルな作り
- 将来他の項目を追加するときも同じパターン(enum + 短文テンプレ)で拡張可能

### 4.3 ログ

- 景気設定値は run metadata に記録(後で比較実験できるように)
- プロンプト実体は LLM 入出力ログ(`*.jsonl`)に含まれるため、別途の記録は不要

## 5. 将来拡張(本フェーズ外)

- 景気の動的変化(ステップ途中で変わる)
- 気候・ニュース・為替など他の環境量
- 設定項目の段階的追加(観察によって「これは決めた方が良い」と判明したもの)
