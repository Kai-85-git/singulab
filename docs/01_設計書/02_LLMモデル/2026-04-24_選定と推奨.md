## 0.5. 設計哲学: 実人間の認知能力に合わせたモデルサイズ

### 本シミュレーションは「頭の良すぎる LLM」を**意図的に使わない**

議事録 §5.3 の 3 階層モデルは **「LLM が人間を超越しないよう、人間のキャパシティ(記憶・認知)の範囲内に収める」** を設計の核に据えている。同じ思想を **モデルサイズ選定** にも適用する。

### モデルサイズと「疑似 IQ」の対応(経験則)

| モデルサイズ | 推論特性 | 疑似的な人間 IQ | オフィス内での立ち位置 |
| --- | --- | --- | --- |
| 70B+(GPT-5 級) | 完璧に articulate、ミスしない、常に合理的 | ~140+ | **現実のオフィスには存在しない**。一人だけ PhD の天才 |
| 20-30B | 洗練された議論、論理的綻び少ない | ~125-135 | 優秀なマネージャー層 |
| **7-8B** | 普通の会話、たまに事実を取り違える | **~100-120** | **一般社員の平均像** |
| **3-4B** | シンプルな推論、繰り返しや勘違いあり | **~85-105** | **現実の職場のリアル多数派** |
| 1-2B | しばしば混乱、短い応答、反復 | ~70-85 | 疲れた日の自分 / 新人 / 眠い人 |

### なぜ 3-4B が「意思決定の創発観察」に最適か

議事録・要件定義から抽出される 5 つの根拠:

1. **兵頭氏アドバイス(議事録 §7)**: 「賢い LLM ほど『頭のいい会話』になりがち、純粋で衝動的な人間の側面が見えない」 → 3-4B は **適度な抜け・感情優位** を自然に示す
2. **役職を設定しない(議事録 §4)**: 自然発生を観察したいのに、70B は勝手に「議論のリーダー役」を完璧に演じてしまう → 小さいモデルは**役割固定化しにくい**
3. **「指示を書きすぎない」(議事録 §4)**: 大型モデルはプロンプトの微小なヒントから完璧に意図を汲み取ってしまう → 小型モデルは**解釈の揺らぎが大きく、人間的**
4. **ダンバー数・認知限界との整合**: 3 階(世界の法則)で人間の記憶制約を課すなら、モデル自体も人間に近い推論能力が自然
5. **フリーライド・諦め・無言の観察**: 賢いモデルは議論を諦めない / 無言にならない。小型モデルは **自然に「もういいや」を出力する**

### 結論

**3-4B モデルは RTX 3060 6GB への妥協ではなく、シミュレーション品質を最大化する積極的選択である。** むしろ本件に 70B を持ち込むと、全員が PhD の企業会議を演じてしまい、**創発すべき人間的ダイナミクスが死ぬ**。

### 発展: IQ の分布を持たせる(オプション)

本物のオフィスには IQ 85〜130 が混在している。エージェントごとに使うモデルサイズを振る案:

| 割合 | モデル | 役割 |
| --- | --- | --- |
| 60% | **Llama 3.2 3B abliterated** 🇺🇸 | 多数派(平均的な社員) |
| 30% | **Qwen3 4B abliterated** 🇨🇳 | やや賢い層(中堅・発言の軸になる人) |
| 10% | **Dolphin 3.0 abliterated 8B** 🇺🇸(CPU オフロード) | 「キレ者」枠(稀にしか発話しない観察者役にしても面白い) |

これにより:
- 「自然発生するリーダー」が推論能力の高いエージェントから出やすいか検証可能
- 100 人大集団でも**全員が 8B だと不自然**、だが**全員が 3B でも単調**
- 「誰が議論をリードするか」の創発が、モデルサイズ分布でどう変わるかを観察できる

→ **2 回目以降の実験**でこの IQ 分布実験を組み込むと研究価値が上がる。初回は均一モデルで OK。

## 1. 結論サマリ(TL;DR)

### 推奨モデル(RTX 3060 Laptop 6GB 最適化)

| 優先度 | モデル | 国籍 | Ollama タグ | VRAM(Q4_K_M) | 用途 |
| --- | --- | --- | --- | --- | --- |
| ★★★ | **Qwen3 4B abliterated** | 🇨🇳 (Alibaba) | `huihui_ai/qwen3-abliterated:4b` | ~2.7GB | **メイン**。日本語◎・小集団も大集団も兼用 |
| ★★★ | **Llama 3.2 3B abliterated** | 🇺🇸 (Meta) | `huihui_ai/llama3.2-abliterated:3b` | ~2.1GB | **100 人大集団の本命**。並列を最大化 |
| ★ (stretch) | **Dolphin 3.0 abliterated 8B (Llama 3.1)** | 🇺🇸 (Meta + Cognitive Computations) | `huihui_ai/dolphin3-abliterated:8b-llama3.1-q4_K_M` | ~5GB (CPU オフロード併用) | 品質検証用・小集団のみ |

> **🚫 日本製モデル(ELYZA / Sarashina / Swallow / Shisa 等)は対象外**。理由: 公式 abliterated 版が存在せず、本プロジェクトの **Uncensored 要件**(議事録 §7)を満たさないため。米(Llama)× 中(Qwen)の 2 軸で進める。日本語品質は Qwen3 がカバー(Shisa.AI のベンチでも開源 8B 級の日本語性能トップ)。

### 最重要な結論

- **8B を 100 並列は無理**(6GB VRAM では物理的に不可能)
- **4B で 4〜8 並列、3B で 8〜12 並列**が現実線
- 100 エージェント × 100 ステップは **4B モデルでバッチ直列(8 並列 × 13 バッチ)** が最も堅い。所要時間 **約 60〜90 分**
- 品質重視の小集団(2〜10 人)だけ 8B ハイブリッドで回し、大集団は 3B/4B で回す **二段構成** を推奨

## 3. VRAM バジェット(6GB 向け)

### 3.1 必要な内訳(単一モデル時)

| 項目 | 6GB VRAM の使い道 |
| --- | --- |
| Windows 11 の GPU 常駐(デスクトップ描画等) | ~0.4GB |
| Ollama ランタイム + CUDA コンテキスト | ~0.3GB |
| **モデル本体に使える最大値** | **~5.3GB** |

### 3.2 モデルごとの並列可能数(num_ctx=2048、Q4_K_M)

| モデル | モデル占有 | 1 seq の KV | 並列 2 | 並列 4 | 並列 8 | 並列 12 |
| --- | --- | --- | --- | --- | --- | --- |
| 3B (Llama 3.2 3B) | 2.1GB | 0.15GB | 2.4GB ✅ | 2.7GB ✅ | 3.3GB ✅ | 3.9GB ✅ |
| 4B (Qwen3 4B) | 2.7GB | 0.2GB | 3.1GB ✅ | 3.5GB ✅ | 4.3GB ✅ | 5.1GB ⚠️ |
| 8B (Llama 3.1 8B) | 5.0GB | 0.4GB | 5.8GB ⚠️ | **OOM** | **OOM** | **OOM** |

→ **3B は 12 並列、4B は 8 並列、8B は 1 並列が 6GB の限界**。KV キャッシュを `q8_0` 量子化すれば若干余裕が出るが、体感上の変化は軽微。

### 3.3 num_ctx(コンテキスト長)の指針

本シミュレーションは「短い状態記述 + 近傍エージェントの発話履歴」程度で、1 エージェント 1 ターンの入力は **2K トークンに収まる** と見込める。

| 設定 | 推奨値 |
| --- | --- |
| `num_ctx` | **2048**(デフォルトの 4096 から半減) |
| `num_predict`(max output) | 256(1 発話は短めに制限) |

これで KV キャッシュを節約し、並列度を稼ぐ。

## 5. 並列実行戦略(RTX 3060 Laptop 6GB 向け)

### 5.1 100 人大集団の実行プラン

**現実解: Llama 3.2 3B abliterated + Ollama 並列 8 + バッチ直列**

```
100 エージェント を 13 バッチ(8 × 12 + 4)に分割
→ 各バッチで同時に 8 エージェントを推論
→ 全バッチ完了で 1 ステップ終了
```

#### 時間見積もり(800,000 tokens 全体)

| 構成 | スループット(集約) | 合計時間 |
| --- | --- | --- |
| 3B + Ollama 並列 8 | ~250〜350 tok/s | **~45〜55 分** |
| 3B + Ollama 並列 12 | ~350〜450 tok/s | **~30〜40 分** |
| 4B + Ollama 並列 4 | ~140〜180 tok/s | ~80〜100 分 |
| 4B + Ollama 並列 8 | ~220〜280 tok/s | ~50〜60 分 |
| 8B CPU+GPU(並列 1) | ~10 tok/s | 約 22 時間(実用外) |

→ **3B で並列 8〜12 が最もバランス良い**。ハッカソン当日に完走可能。

### 5.2 小集団(2〜10 人)の実行プラン

時間に余裕があるので **品質重視 8B** が使える:
- Ollama + Dolphin 3.0 abliterated 8B(GPU+CPU オフロード)
- 並列 1、単純直列で実行
- 10 人 × 100 ステップ = 1000 呼び出し × 80 tokens = 80K tokens → **約 90 分**

### 5.3 vLLM / llama-server は?

6GB VRAM だと **vLLM のメリットが薄い**:
- vLLM は最低 2〜3GB の overhead がある
- モデル + KV + overhead で 6GB を超えやすい
- Ampere の FP8 非対応も相まって、Ollama と大差ない結果になりがち

→ **本機では Ollama で素直に運用するのが最適**。vLLM は「デスクトップ 3060 12GB 以上 + 大量並列」でこそ真価を発揮する。

### 5.4 環境変数(Windows / Ollama)

```powershell
# システム環境変数に設定
$env:OLLAMA_NUM_PARALLEL = 8         # 3B の場合。4B なら 4-6
$env:OLLAMA_FLASH_ATTENTION = 1      # 必須
$env:OLLAMA_KV_CACHE_TYPE = "q8_0"   # KV を 8bit 量子化で節約
$env:OLLAMA_KEEP_ALIVE = "10m"       # モデル切替を減らす
$env:OLLAMA_MAX_LOADED_MODELS = 1    # メモリ圧迫を避ける
```

### 5.5 ノート PC 特有の注意

| 項目 | 対策 |
| --- | --- |
| Thermal throttle | 冷却台使用、ファン最大(Armoury Crate の Turbo モード)、筐体背面を浮かす |
| 電源 | **必ず AC 接続**。バッテリ駆動だと TGP が 30W に絞られ、半分以下の速度に |
| メモリ | Chrome / VS Code 等を閉じる。OS 実使用を 4GB 以下に抑える |
| 長時間稼働 | 1 時間ごとに 5 分休憩を入れるとセンサー保護が効きやすい |

## 6. 代替案: API ハイブリッド構成

100 人大集団をローカルで回すのが時間的に厳しい、または品質に不満がある場合:

### 構成
- **小集団 + 品質検証**: ローカル(Qwen3 4B abliterated)で観察
- **大集団本番**: API 系を併用
  - **Claude Haiku 4.5**: 日本語強い、速い、価格も妥当($1/M token 台)
  - **Gemini 2.5 Flash**: 最安($0.1/M token 台)、日本語可
  - **Grok-4**: 比較的ガード弱め(議事録 §Q&A 言及)、日本語も使える

### コスト試算(800K tokens / 100 人 × 100 ステップ)

| モデル | 入力単価(USD/M) | 出力単価 | 概算総額(入出力半々) |
| --- | --- | --- | --- |
| Claude Haiku 4.5 | $1 | $5 | **約 $2.4**(約 360 円) |
| Gemini 2.5 Flash | $0.1 | $0.4 | **約 $0.2**(約 30 円) |
| GPT-4o mini | $0.15 | $0.6 | **約 $0.3**(約 45 円) |

→ **1 回の実験なら数十円〜数百円**。ハッカソン当日の本番だけ API に逃がすのは合理的。

### リスク
- **API 系は uncensored ではない**(ガードあり)→ 創発観察の趣旨とズレる
- Grok-4 のみ比較的ガードが弱い
- 提出資料での「uncensored 観察」主張と整合しなくなる

→ API は「スループット確保のバックアッププラン」として位置づけ、本線は **ローカル 3B/4B abliterated** を推奨。

---

## 7. リスクと回避策

| リスク | 回避策 |
| --- | --- |
| **VRAM OOM** | `num_ctx=2048`、`num_predict=256`、`KV_CACHE_TYPE=q8_0`、並列数を段階的に上げて検証 |
| **時間不足** | モデルを 4B→3B に落とす、ステップを 100→50 に削減(議事録は「100 程度」で幅あり)、API 併用 |
| **Thermal throttle** | AC 接続 + 冷却台 + Turbo モード、1 時間に 5 分休憩 |
| **ノート PC 特有の GPU TGP 絞り** | Armoury Crate で Turbo プロファイル、AC 必須 |
| **abliteration 品質低下** | 元モデルで同条件を並列比較、出力崩壊と意味のある感情表現を区別 |
| **再現性** | `temperature=0.2〜0.4`、`seed` 固定、3〜10 回繰り返しの分布評価(議事録 §Q&A) |
| **倫理・安全** | Uncensored は研究用途限定。Python 側で NG ワードフィルタを挟む |
| **ライセンス** | 提出物 README にベース + abliterator を明記 |
