# シミュレーションループ(たたき)

## 1. 1 ステップの実行順序

参考実装の 4 フェーズ構成を踏襲しつつ、環境量(2階)・物理法則(3階)・汎用イベントの扱いを追加する。

```
[Step t に入る]
 ├─ (0) イベント活性化判定
 │       └─ 未活性の Event を start_step に達していれば activate()
 ├─ (1) 物理法則の進行(3階)
 │       └─ 各 Physics.step(environment, dt)
 │          (拡散 / 減衰 / 反応 等で Environment を更新)
 ├─ (2) Agent state 更新
 │       └─ 各 Agent の in_place / current_place を位置から再計算
 ├─ Phase 1: Message 決定(LLM 呼び出し)
 │       └─ 各 Agent.decide_message(
 │             place_status, env_readings, nearby_agents, event_infos, step)
 │          (位置情報は含めない)
 ├─ Phase 2: Message 送信
 │       └─ can_communicate() を満たす他 Agent に receive_message()
 │          messages.jsonl に 1 行追記
 ├─ Phase 3: Action 決定(LLM 呼び出し)
 │       └─ 各 Agent.decide_action(
 │             place_status, env_readings, nearby_agents, event_infos,
 │             message_to_send, step)
 │          memory_reasoning.jsonl に 1 行追記(バッチ書き込み)
 ├─ Phase 4: 移動実行
 │       └─ action == "move" なら direction へ 1 セル移動
 ├─ (5) Agent state 再更新(移動後)
 ├─ (6) 統計収集(占有率・メッセージ数・イベント半径内人数 等)
 └─ (7) 可視化 / ログ書き出し
[Step t 終了]
```

## 2. 各フェーズの詳細

### Phase 0: イベント活性化

```python
for event in self.events:
    if not event.is_active(self.step) and event.should_activate(self.step):
        event.activate(self)
        self._log_event("activated", event, self.step)
```

- 参考実装の `fire_configs / fire_states` の挙動を抽象化したもの
- `should_activate(step)`: 典型的には `step >= start_step` かつ未発生
- 発生時に位置がランダムなら `_generate_random_position()` を使う
- `events.jsonl` に発生ログを書く

### Phase 1 の直前: 物理進行(MVP では空でも可)

```python
for law in self.physics_laws:
    law.step(self.environment, dt=self.dt)
```

- 2階の環境量(例: 飛沫濃度)を 3階の法則(例: 拡散)で更新
- MVP ではこのブロックは空実装(参考実装と等価なシナリオで必要ないため)

### Phase 1 / 2 / 3 / 4: 参考実装と同じ

**重要**: メッセージ送信は **移動前の位置関係** に基づく(参考実装同様)。
この順序により、「話してから動く」という人間的な流れになる。

### Phase 5〜7: 統計・ログ

統計は少なくとも以下を収集:

| 項目 | 計測タイミング | 出力先 |
| --- | --- | --- |
| 全体占有率(`agents_in_place / num_agents`) | Phase 5 後 | `stats`, `statistics.png` |
| 各場所の占有率 | Phase 5 後 | 同上 |
| 各イベント半径内の人数 | Phase 5 後 | 同上 |
| メッセージ送信数 | Phase 2 | `metrics.jsonl` |
| LLM 呼び出し回数・所要時間 | Phase 1 / 3 | `simulation.log` |

## 3. タイミング設計の要点

### 3.1 なぜ「Message → Action」の順か(参考実装踏襲)

- 先にメッセージを決めることで、「発話内容を踏まえた行動」が可能になる
- 例: 「逃げよう」と言った → 自分も動く / 「ここ安全」と言った → 自分は残る
- 逆順(Action 先)だと、発話が行動と整合しにくくなる

### 3.2 位置情報の露出制御

| Phase | 位置情報 | 理由 |
| --- | --- | --- |
| Phase 1(Message) | **非公開** | 位置をメッセージに直接書かせない(情報伝達を創発させるため) |
| Phase 3(Action) | **公開** | 移動判断に必要 |

### 3.3 イベント知覚の一貫性

- Phase 1 と Phase 3 の両方で、同一の `event_infos` を渡す
- つまり「メッセージは書くが、行動判断では知らない」のような不整合は発生しない

## 4. 擬似コード(Orchestrator)

```python
class Simulation:
    def step_simulation(self):
        self.step += 1

        # (0) イベント活性化
        for ev in self.events:
            if not ev.is_active(self.step) and ev.should_activate(self.step):
                ev.activate(self)
                self._log_event(ev)

        # (1) 物理進行
        for law in self.physics_laws:
            law.step(self.environment, dt=self.dt)

        # (2) Agent 状態更新
        for a in self.agents:
            a.update_state(self.places)

        # Phase 1: Message 決定
        message_decisions = []
        for a in self.agents:
            nearby = a.get_nearby_agents(self.agents)
            ps = self.get_place_status(a.current_place) if a.in_place else None
            env = self.environment.readings_at(a.position)
            evs = [ev.perceived_by(a) for ev in self.events if ev.is_active(self.step)]
            evs = [e for e in evs if e is not None]
            md = a.decide_message(ps, env, nearby, evs, self.step)
            message_decisions.append((a, md, nearby))

        # Phase 2: Message 送信
        for a, md, nearby in message_decisions:
            content = md.get("message", "")
            if content and nearby:
                for other in nearby:
                    other.receive_message(a.id, content, step=self.step)
                    self._log_message(a.id, other.id, content, md.get("reasoning", ""))

        # Phase 3: Action 決定
        action_decisions = []
        memory_records = []
        for a, md, nearby in message_decisions:
            ps = self.get_place_status(a.current_place) if a.in_place else None
            env = self.environment.readings_at(a.position)
            evs = [ev.perceived_by(a) for ev in self.events if ev.is_active(self.step)]
            evs = [e for e in evs if e is not None]
            ad = a.decide_action(ps, env, nearby, evs, md.get("message", ""), self.step)
            action_decisions.append((a, ad))
            memory_records.append({
                "step": self.step, "id": a.id,
                "memory": ad.get("memory", ""),
                "reasoning": ad.get("reasoning", "")
            })
        self._log_memory_reasoning_batch(memory_records)

        # Phase 4: 移動
        for a, ad in action_decisions:
            if ad["action"] == "move" and ad["direction"]:
                a.move(ad["direction"])

        # (5)(6)(7)
        for a in self.agents:
            a.update_state(self.places)
        self._collect_stats()
        self._maybe_visualize()
```

## 5. 終了条件

MVP は参考実装と同じく `step < duration` でループを抜ける。

将来的に以下を検討:

| 終了条件 | 判定 |
| --- | --- |
| エージェント全滅 | 生存 Agent 数が 0 |
| 目的達成 | 例: 「全員が場所 X に集まった」等のシナリオ特有条件 |
| メトリクス収束 | 例: 占有率の移動平均が N ステップ変動しない |

## 6. 例外・エラー処理方針

| 事象 | 対応 |
| --- | --- |
| LLM 接続失敗(Ollama 停止) | ステップ開始前にチェック、失敗ならユーザ向けメッセージを出して終了 |
| LLM 呼び出しタイムアウト | 個別 Agent を `stay` にフォールバック、`simulation.log` に warning |
| JSON パース失敗 | テキストパースへフォールバック(参考実装踏襲) |
| 位置境界違反 | `move()` 内でクランプ(参考実装踏襲) |
| イベント未登録型 | 起動時バリデーションでエラー |
