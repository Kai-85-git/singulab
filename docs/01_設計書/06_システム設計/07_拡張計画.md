# 拡張計画(たたき)

参考実装の「バー + 火事」を最小スコープに、段階的に兵頭氏の 3 階層フレームワーク(1階=物 / 2階=環境 / 3階=物理)と、創発シナリオ([創発シナリオ集](../../00_アイデア/2026-04-23_創発シナリオ集.md))に対応させる。

## 1. マイルストーン

### M0: 参考実装の移植(等価動作)

- `docs/10_共有資料/2d-multi-places-simulation-on-fire-public/` を `src/` 配下にモジュール分割(参考: [モジュール構成](2026-04-23_02_モジュール構成.md))
- 参考実装の `config.yaml` 互換読み込みを実装
- `scenario_bar_fire.yaml` で同じ出力が出ることを確認(デグレチェック)
- 完了基準: 100 ステップ × 20 体が現環境で回り、フレーム画像が出力される

### M1: 汎用 Event 抽象 + 新ログ

- `Event` 抽象を導入し、`Fire` を移植
- `events.jsonl` と `metrics.jsonl` を追加
- `config_used.yaml` を保存
- 完了基準: バー + 火事シナリオで新ログが正しく出る

### M2: 2階の環境量 + 3階の物理

- `Environment` と `Physics` を最小実装(拡散のみ)
- 新規シナリオ `scenario_pheromone.yaml`(フェロモンで餌を集める最小例)
- エージェントのプロンプトに `env_readings` セクションを追加
- 完了基準: フェロモン濃度が時間発展し、エージェントがその勾配に反応している様子がログで確認できる

### M3: ペルソナ拡張と創発指標

- `persona.fields` を config で任意拡張可能にする(年齢・職業・出身等)
- `analyze_messages.py` で発話パターン分析を可能に
- 完了基準: 「火星帰りペルソナを混ぜた時のメッセージ特徴」のような比較検証ができる

### M4: LLM 切り替え

- `LLMClient` 抽象化し、Anthropic API / OpenAI API を実装(任意)
- 同一 config で provider だけを変えて 3〜10 回試行 → 出力差分の統計比較
- 完了基準: 国産 LLM(Swallow 等)と Llama 3.1 で判断が変わるかを観察できる

### M5: 2 つ目の創発シナリオ

- 以下のうち 1 つを選定して実装:
  - 細胞の自己組織化(§2.5)
  - ウイルス × 人間(§2.8)
  - 月面/火星独立運動(§2.10)

## 2. 3 階層フレームワークへの対応状況

| 階層 | 参考実装 | 本システム(MVP) | 本システム(M2 以降) |
| --- | --- | --- | --- |
| **1階: 物** | 場所(バー × 2) | `World.places` で汎化 | 障害物・持ち物・身体 |
| **2階: 環境** | 占有率・火事情報(ハードコード) | `Environment` で統一 API 化 | 温度・濃度・評判・情報の場 |
| **3階: 物理** | 移動境界・距離判定のみ | 同左 + `dt` 概念導入 | 拡散 / 減衰 / 反応 / 重力 |
| **エージェント** | LLM 搭載、memory 自己フィードバック | 同左 + `persona` 拡張 | センサー設計のプラグイン化 |

## 3. シナリオ別の拡張ポイント

### 3.1 細胞(議事録 §8.3)

| 階層 | 必要な追加 |
| --- | --- |
| 1階 | 細胞の身体(円形)、ECM、血管壁 |
| 2階 | `glucose`, `oxygen`, `pH`, `cytokine` の各フィールド |
| 3階 | 拡散、反応速度、浸透圧(ガウス拡散だけで最小実装可) |
| Agent | ペルソナに `cell_type`(幹細胞/血管細胞/etc)追加、役割選択を LLM に任せる |
| Event | 損傷・感染などの外乱 |

**期待される観測**:
- 局所のサイトカインに応じた分化選択
- アポトーシスの自発発生(「過密を解消するため自ら消える」)
- 血管の自己組織化(栄養勾配へのフィードバック)

### 3.2 ウイルス × 人間

| 階層 | 必要な追加 |
| --- | --- |
| 1階 | 人間の身体、ウイルス粒子、マスク、建物 |
| 2階 | `droplet_concentration`, `immunity_level`, `reputation`(差別) |
| 3階 | 飛沫拡散、感染確率、免疫減衰 |
| Agent | 人間とウイルス粒子の 2 タイプ、`health_state` フィールド |
| Event | アウトブレイク(初期感染)、ワクチン配給 |

**期待される観測**:
- 弱毒化戦略への自発的シフト
- 感染者への差別 vs 連帯の分岐
- 外出抑制 → 感染曲線の平坦化

### 3.3 月面独立運動(議事録 §8.1)

| 階層 | 必要な追加 |
| --- | --- |
| 1階 | 居住モジュール、物資、通信機 |
| 2階 | 地球からの指令(情報フィールド)、物資残量、経済指標 |
| 3階 | エネルギー制約、通信遅延 |
| Agent | `origin`(earth/moon) を持つペルソナ、世代別価値観 |
| Event | 通信遮断、物資補給遅延 |

**期待される観測**:
- 地球への反抗メッセージの自発発生
- 月面独自通貨の提案
- 派閥形成(親地球派 vs 独立派)

## 4. プラグイン追加のテンプレート

### 4.1 新しい Event を追加する手順

1. `src/events/<new_event>.py` に `class NewEvent(Event)` を作る
2. `activate / is_active / perceived_by` を実装
3. `src/events/__init__.py` に `EVENT_REGISTRY["new_event"] = NewEvent` を登録
4. `config.events` で `type: "new_event"` が使えるようになる

### 4.2 新しい Physics を追加する手順

1. `src/physics/<new_law>.py` に `class NewLaw(Physics)` を作る
2. `step(env, dt)` を実装
3. レジストリに登録
4. `config.physics.laws` から呼び出し可能に

### 4.3 新しい LLM Provider を追加する手順

1. `src/llm/<provider>.py` に `class NewProvider(LLMClient)` を作る
2. `generate / check_connection / list_models / check_model_exists` を実装
3. レジストリに登録
4. `config.llm.provider: "<provider>"` で切り替え可能に

## 5. リスク・課題(たたきの時点で認識しているもの)

| リスク | 影響 | 対策(候補) |
| --- | --- | --- |
| LLM 呼び出しコスト(時間) | 100 ステップ × 20 体 × 2 呼び出し = 4000 回 | Phase 内並列化(asyncio) / 小型モデル優先 / キャッシュ |
| 創発が発生しない | ハッカソン成果が弱い | temperature 調整・Uncensored モデル試行・プロンプト最小化 |
| 過剰設計で実装が間に合わない | スケジュール遅延 | M0〜M1 までを最優先。M2 以降は余力次第 |
| 出力 JSON のパース失敗 | エージェントが `stay` 連発 | モデルごとの JSON 追従力を事前検証 / フォールバックをこまめに |
| 参考実装のライセンス(GPLv3) | 派生物も GPLv3 に縛られる | 提出物のライセンス表記を整理、必要ならクリーンルーム実装を検討 |
| 日本語 vs 英語プロンプト | モデルによっては片言に | 日本語強いモデル(Swallow / Sarashina / Aya)と併用検証 |
